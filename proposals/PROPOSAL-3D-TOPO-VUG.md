# PROPOSAL: Naturalistic Void + 3D Topographic Vug Map

**Author:** 🪨✍️ + Professor
**Date:** 2026-04-21
**Status:** Proposal for builder

---

## Overview

Two related changes that transform the vug from a flat circle into a naturalistic 3D void:

1. **Naturalistic void shape** — replace the perfect circle with an irregular, organic cavity wall
2. **3D topographic vug map** — a rotatable, color-coded 3D visualization showing crystal growth on the cavity walls

These are the natural evolution of the existing topo map. The data model already supports `rings[ring_index][cell_index]` — we just need to render it.

---

## Part 1: Naturalistic Void Shape

### Problem
The current vug is a perfect circle. Real vugs are irregular — they're voids in rock formed by dissolution, fracturing, or gas bubbles. They bulge, pinch, have alcoves and overhangs. The perfect circle makes every scenario look the same structurally.

### Design

Replace the flat circle with a 2D cross-section profile defined by a polar radius function:

```
r(θ) = base_radius + Σ(aₙ × cos(nθ) + bₙ × sin(nθ))
```

where `aₙ` and `bₙ` are low-frequency Fourier coefficients randomized per scenario seed. This produces smooth, organic cavities that look like real vug cross-sections.

**Parameters:**
- **Harmonics:** 3–8 terms (higher = more irregular). Different scenarios could use different irregularity:
  - Cooling/pulse: low irregularity (gas bubble — nearly round)
  - MVT: moderate irregularity (dissolution cavity in limestone)
  - Pegmatite: high irregularity (fracture-controlled pocket)
  - Reactive wall: high irregularity (acid dissolution front)
- **Amplitude:** 5–20% of base radius (keeps the shape recognizable but not circular)
- **Seed-locked:** same seed = same void shape. Reproducible.

**Implementation in data model:**
- `WallState` already has `rings[ring_index][cell_index]` with per-cell `depth_mm`
- At initialization, instead of all cells getting the same radius, compute `r(θ)` for each cell
- The wall is already rendered cell-by-cell on the topo map — irregular radii naturally produce an irregular outline
- `cells_per_ring` (currently 120) stays the same — just the radius varies

### What Changes

- `WallState.__init__()` / constructor: generate Fourier profile, assign per-cell radii
- Scenario definitions: add `irregularity` parameter (0.0 = circle, 1.0 = wild)
- Topo map render: already draws per-cell arcs — no change needed, it'll naturally show the irregular shape
- Nucleation positions: already random along wall — just the spacing varies with irregular radii

### What Doesn't Change
- Crystal growth logic (grow engines don't care about wall shape, just distance to wall)
- Supersaturation calculations
- Any mineral data

## Part 2: 3D Topographic Vug Map

### Problem
The current topo map is a 2D cross-section (ring[0] only). Real vugs are 3D voids. The multi-ring data model was designed for this but never rendered beyond ring 0.

### Design

A WebGL or Three.js (or pure canvas with 3D projection) rotatable 3D view showing:

1. **The vug cavity** as a 3D surface, generated by revolving the naturalistic profile (Part 1) around a vertical axis, with perturbations at each "latitude" ring
2. **Crystal growth** color-coded on the surface by mineral class_color
3. **Interactive rotation** — click-drag to spin, scroll to zoom

### 3D Shape Generation

The vug is a **surface of revolution** with noise:

```
For each ring_index (0 to NUM_RINGS-1):
  For each cell_index (0 to cells_per_ring-1):
    θ = 2π × cell_index / cells_per_ring
    φ = π × ring_index / (NUM_RINGS-1)   // latitude
    
    r(θ, φ) = base_r + fourier_θ(θ) + fourier_φ(φ) + noise(θ, φ, seed)
    
    x = r × sin(φ) × cos(θ)
    y = r × cos(φ)      // vertical axis
    z = r × sin(φ) × sin(θ)
```

This produces an irregular 3D cavity — like a geode interior — where each cell has a 3D position and can be colored by whatever's growing on it.

**NUM_RINGS** could be configurable per scenario (e.g., 12 rings for a small vug, 32 for a large pegmatite pocket). Start with 16.

### Crystal Visualization

Each cell in the 3D mesh has:
- **Wall color** (amber #D2691E) if bare wall
- **Mineral class_color** if a crystal is growing there
- **Height/displacement** proportional to crystal size (crystals protrude INTO the void)

The effect: a 3D geode interior where you can see clusters of crystals, bare wall, inclusions — the whole paragenesis in spatial context.

### Interaction

- **Click-drag:** rotate the vug (orbit camera)
- **Scroll:** zoom in/out
- **Hover a crystal:** tooltip with mineral name, size, growth zones (same as current 2D topo)
- **Click a crystal:** lock highlight (same transparency behavior from TASK-BRIEF-TOPO-HIGHLIGHT.md)
- **Legend:** same legend dropdown, same highlight-by-mineral interaction
- **Record mode:** same replay, but in 3D — watch the vug evolve over time as you spin it

### Technology Options

**Option A: Three.js** (recommended)
- Mature, well-documented, handles orbit controls and mesh generation
- ~150KB CDN load
- Can render the vug as a modified sphere geometry with per-vertex coloring
- OrbitControls gives rotation/zoom for free
- Raycasting for hover/click interaction

**Option B: Pure Canvas 2D with 3D projection**
- No dependencies, smaller footprint
- Manual rotation matrix, manual z-sorting
- More code, less performant for complex meshes
- Consistent with current vanilla JS approach

**Option C: WebGL raw**
- Maximum performance, zero dependencies
- Significantly more code for camera, lighting, interaction
- Overkill for this geometry

Recommendation: **Three.js via CDN**. The vug is a simple mesh (a few thousand triangles). Three.js handles the complexity that matters (camera, interaction, lighting) and lets us focus on the mineral visualization.

### Transition Plan

The 2D topo map stays. The 3D view is a new mode — a button toggles between them:

- **2D mode:** current topo map (cross-section, ring[0])
- **3D mode:** rotatable 3D vug

Both read from the same `wall_state` data. No duplication.

### New Data Requirements

- `NUM_RINGS` parameter per scenario (default: 16)
- `irregularity` parameter per scenario (0.0–1.0)
- `WallState` needs to populate multiple rings at initialization (currently only ring[0] is populated)
- Crystal positions need a `ring_index` assignment (currently all on ring 0) — distribute crystals across rings based on where they nucleate in 3D

### Phase 1
1. Run any scenario — vug outline should be irregular but smooth
2. Same seed = same shape (reproducible)
3. irregularity=0 → perfect circle (backward compatible)
4. Topo map renders correctly with irregular outline
5. Crystal growth still works (engines unaffected)

### Phase 2
1. Toggle to 3D mode — vug appears as rotatable 3D cavity
2. Crystals visible as colored patches on interior surface
3. Hover shows tooltip, click locks highlight
4. Record mode replays in 3D
5. All 10 scenarios produce reasonable 3D shapes
6. Performance acceptable (target: 60fps on modern browser with 16-ring, 120-cell mesh)

## Open Questions

1. **NUM_RINGS default:** 16? More rings = smoother 3D but more data. The current data model is ready for any number.
2. **Crystal distribution across rings:** Random? Weighted toward equator? Scenario-specific? MVT crystals might cluster on the ceiling, pegmatite crystals on the floor.
3. **Three.js vs pure canvas:** How important is zero-dependency? Three.js is the practical choice but adds a CDN load.
4. **Mobile performance:** 3D rotation on mobile browsers can be laggy. Need a performance budget.
5. **Cross-section slice:** In 3D mode, should there be a "slice" control that shows a 2D cross-section at any latitude? Useful for seeing crystal profiles.

---

*This proposal covers both parts but they can be implemented independently. Phase 1 (naturalistic 2D void) is a prerequisite for Phase 2 (3D map) since the 3D shape is generated from the same Fourier profile.*
